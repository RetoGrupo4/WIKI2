Ramas:

Se crean 3 ramas para trabajar:

  1. Master: Principal (usada para versiones finales)
  2. WIP: Proceso (usada para el trabajo continuo, puede contener errores leves)
  3. FIX: Errores (usada para corregir errores graves)

Las ramas WIP y FIX son las utilizadas por los miembros del grupo para trabajar.



Recursos usados:

  1. Propia web de GITHUB.
  2. GITHUB Desktop. (aplicación de escritorio)
  3. GIT BASH. (Consola de comandos específica para GITHUB).
    1. Esto nos permite usar la consola de comandos del propio entorno de desarrollo
  4. Consola de comandos propia del S.O


Comandos:

  1. GIT COMMIT
    1. El contenido se muestra sección de BRANCH en la que se haya subido.
    2. De esta manera se puede ver que archivos han sido actualizados por el nombre del commit.
  2. GIT COMMIT + GIT TAG
    1. El contenido se muestra en la sección de TAGS.
    2. Las versiones correspondientes se clasifican con las etiquetas.
      1. Crea un archivo con el nombre del TAG en una carpeta oculta vinculada al github.
      2. Los TAGS crean ramas propias.
    3. De esta manera tambien se puede ver que archivos han sido actualizados por el nombre del commit.


Control de versiones mediante TAGS

Con el uso del comando GIT TAG podemos llevar un mayor control de las versiones, además de poder mostrarla en el propio proyecto.

Al vincular el proyecto con GITHUB con el comando GIT INIT se genera una carpeta oculta llamada .GIT, la cual nos permite poder acceder a los TAGS mediante la carpeta .GIT\REFS\TAGS

Para mostrar en la web cual es el último TAG creado, a través de un archivo php obtenemos un array de todos los TAGS, en el cual ordenamos de mayor a menor, siendo la posición 0 (la versión mayor) la que cogeremos para mostrarla en la web.
