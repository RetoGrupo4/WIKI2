Ramas:



Se crean 3 BRANCH (Ramas) para trabajar:


Master:

Esta rama se crea por defecto al crear el repositorio. Es usada para versiones finales y funcionales, y son las que se entregan


WIP:

Esta rama se crea una vez creado el repositorio. Es usada para trabajo en proceso y puede contener errores que no entorpezcan al desarrollo del proyecto.


FIX:

Esta rama se crea una vez creado el repositorio. Es usada para correccion de errores graves que impidan continuar con el proyecto.


Las ramas WIP y FIX son las utilizadas por los miembros del grupo para trabajar.


Ademas con el comando GIT TAG podemos crear mas ramas categorizadas por etiquetas de verison y nos permite introducir en un archivo los datos de autor, fecha y hora de creacion...



Recursos usados:



1. Propia web de GITHUB.

  • Limitaciones de tamaño. Los archivos no pueden ser mayores a 100 MB mientras que el repositorio puede alojar 1 GB de información.

    • 2. GITHUB Desktop. (aplicación de escritorio)

  • Sin limite de tamaño a la hora de subir el proyecto al repositorio
  • Esto nos permite subir de manera mas rapida y visual el proyecto a la rama correspondiente
  • Nos facilita la incorporación de descripción
  • Usa de fondo GIT BASH




    • 3. GIT BASH. (Consola de comandos específica para GITHUB).

  • Sin limite de tamaño a la hora de subir el proyecto al repositorio
  • Esto nos permite usar la consola de comandos del propio entorno de desarrollo.
  • Nos permite cambiar entre las ramas con el comando "git checkout <rama>", ademas de poder subir las versiones con las etiquetas




    • 4. Consola de comandos propia del S.O

  • Sin limite de tamaño a la hora de subir el proyecto al repositorio
  • Mismas funcionalidades que el GIT BASH


    • Comandos:



    GIT CHECKOUT <RAMA>

  • Nos permite cambiar de rama en la consola de comandos o en el GIT BASH


    • GIT COMMIT

  • El contenido se muestra sección de BRANCH en la que se haya subido.
  • De esta manera se puede ver que archivos han sido actualizados por el nombre del commit.


    • GIT COMMIT + GIT TAG

  • El contenido se muestra en la sección de TAGS.
  • Las versiones correspondientes se clasifican con las etiquetas.
  • Crea un archivo con el nombre del TAG en una carpeta oculta vinculada al github.
  • Los TAGS crean ramas propias.
  • De esta manera tambien se puede ver que archivos han sido actualizados por el nombre del commit.
  • Se puede añadir mas informacion a la version con GIT TAG -A <version> -m <comentario>.
  • Con el comando GIT SHOW <version> podemos ver la información guardada con el comando anterior.





    • Control de versiones:



    Dependiendo de que recurso estemos usando nos aparece la diferencia de commits respecto a la rama master.



    En el Visual Studio Code aparece que hay 2 commits más, mientras que en la web aparecen 5 commits de diferencia. Esto se debe a que al introducir el comando GIT TAG sobre la rama WIP, el propio VSC solo detecta los commits que se han hecho en esa rama mientras que la web tambien detecta los Tags que se han creado sobre esa rama.







    Con el uso del comando GIT TAG podemos llevar un mayor control de las versiones, además de poder mostrarla en el propio proyecto.



    Al vincular el proyecto con GITHUB con el comando GIT INIT se genera una carpeta oculta llamada .GIT, la cual nos permite poder acceder a los TAGS en la carpeta .GIT\REFS\TAGS



    Para mostrar en la web cual es el último TAG creado, a través de un archivo php obtenemos un array de todos los TAGS, en el cual ordenamos de mayor a menor, siendo la posición 0 (la versión mayor) la que cogeremos para mostrarla en la web.
